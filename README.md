# 📉 Telco Customer Churn Prediction

A machine learning project that predicts whether a telecom customer is likely to churn, with an interactive **Streamlit web app** that gives real-time predictions, a risk category, and a **SHAP-based explanation** of *why* the model made that prediction.

**Live App:** [Streamlit App](https://telco--customer-churn-predictor.streamlit.app)
**Dataset:** [IBM Telco Customer Churn Dataset](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)

## 🔍 Project Workflow

The full analysis lives in `Customer_churn.ipynb`. It followed this pipeline:

### 1. Data Cleaning
- Converted `TotalCharges` to numeric (had blank strings for new customers) and filled missing values with 0
- Dropped `customerID` (identifier, no predictive value)
- Standardized inconsistent category labels (e.g. `"No internet service"` / `"No phone service"` → `"No"`)
- Encoded target: `Churn` → 1 (Yes) / 0 (No)

### 2. Exploratory Data Analysis (EDA)
Key patterns found in the data:
- **Contract type is the single strongest churn driver** — month-to-month customers churn far more than one/two-year contract customers.
- **Tenure matters heavily** — customers in their first few months are much more likely to churn; loyalty builds over time.
- **Payment method** — customers paying by **electronic check** churn more than those on automatic bank transfer/credit card.
- **Fiber optic internet customers churn more** than DSL customers, despite fiber being the "premium" service — likely tied to pricing or service issues.
- **Add-on services reduce churn** — customers with Online Security, Tech Support, etc. are stickier.
- **Class imbalance**: ~73% stayed vs ~27% churned — handled with SMOTE during training.

### 3. Feature Engineering
- `num_addons` — count of add-on services a customer has (Online Security, Backup, Device Protection, Tech Support, Streaming TV, Streaming Movies)
- `charge_per_addon` — `MonthlyCharges / (num_addons + 1)`, a normalized spend-per-service metric

### 4. Preprocessing Pipeline
- **Numerical features**: median imputation + `StandardScaler`
- **Categorical features**: most-frequent imputation + `OneHotEncoder`
- Combined via `ColumnTransformer`

### 5. Model Comparison
Three models were trained and compared using GridSearchCV:

| Model | ROC-AUC |
|---|---|
| Logistic Regression | ~0.84 |
| Random Forest | ~0.843 |
| **XGBoost (selected)** | **~0.845** |

**XGBoost was selected** as the final model — best ROC-AUC, and naturally handles the class imbalance via `scale_pos_weight`, paired with SMOTE inside the training pipeline for further balance.

### 6. Final Tuned Model
```
XGBClassifier(
    max_depth=3,
    n_estimators=200,
    learning_rate=0.05,
    subsample=0.8,
    colsample_bytree=0.8,
    scale_pos_weight=<class ratio>,
    eval_metric="aucpr"
)
```
Hyperparameters chosen via `GridSearchCV`.

### 7. Threshold Tuning
Instead of the default 0.5 cutoff, a **custom decision threshold of 0.45** was chosen after analyzing the precision/recall tradeoff — since in churn prediction, **missing a customer who will churn (false negative) is usually costlier** than flagging a loyal customer for retention outreach (false positive).

**Test set performance at threshold = 0.45:**

| Metric | Churn class (1) |
|---|---|
| Precision | 0.45 |
| Recall | 0.89 |
| F1-score | 0.60 |
| Support | 374 |

| Overall | Value |
|---|---|
| Accuracy | 0.68 |
| Macro avg F1 | 0.67 |
| Weighted avg F1 | 0.70 |

This threshold intentionally favors **recall over precision** — catching 89% of customers who will actually churn, at the cost of some false alarms — which fits a retention-focused business use case better than a default 50% cutoff.

## 🖥️ The Streamlit App

`app/app.py` provides:
- **A form** to enter a customer's profile (demographics, contract, services, billing)
- **Churn probability** — the model's predicted likelihood of churn
- **Prediction** — Will churn / Will stay, based on the 0.45 threshold
- **Risk category** — a color-coded tier for quick triage:

| Probability | Category |
|---|---|
| < 20% | 🟢 Low Risk |
| 20–45% | 🟡 Medium Risk |
| 45–70% | 🟠 High Risk |
| ≥ 70% | 🔴 Very High Risk |

- **SHAP explanation** — a bar chart showing the top factors driving *that specific customer's* prediction (red = pushes toward churn, green = pushes toward staying), so predictions aren't a black box.

## ⚙️ Running Locally

```bash
# 1. Clone the repo
git clone https://github.com/Abhi-aryaa/telco_customer_churn.git
cd telco_customer_churn

# 2. Create & activate a virtual environment
python -m venv venv
venv\Scripts\activate        # Windows
source venv/bin/activate     # Mac/Linux

# 3. Install dependencies
pip install -r requirements.txt

# 4. Train the model (creates Model/churn_model.joblib)
python Model/train_model.py

# 5. Run the app
cd app
streamlit run app.py
```

The app opens at `http://localhost:8501`.

## ☁️ Deployment (Streamlit Community Cloud)

1. Push the repo to GitHub (including the generated `churn_model.joblib`)
2. Go to [share.streamlit.io](https://share.streamlit.io) → sign in with GitHub
3. Click **Create app** 
4. Set:
   - **Repository**: `yourusername/telco_customer_churn`
   - **Branch**: `main`
   - **Main file path**: `app/app.py`
5. Click **Deploy**

Streamlit Cloud installs `requirements.txt` and launches the app automatically. Any future `git push` to `main` redeploys it.

## 🧠 Business Insights & Recommendations

Based on the model and EDA findings:
- **Target month-to-month customers early** — offering incentives to upgrade to annual contracts is likely the single highest-leverage retention lever.
- **Focus retention efforts in the first 3–6 months of tenure** — this is the highest-risk window.
- **Investigate electronic check and fiber-optic customer experience** — both segments show disproportionately high churn, suggesting friction in billing or service quality worth addressing directly.
- **Bundling add-on services** (security, tech support) correlates with stickier customers — worth testing as a proactive retention offer for at-risk segments identified by the model.
